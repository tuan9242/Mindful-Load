V. CAM KẾT VỀ QUY TRÌNH LÀM VIỆC (WORKFLOW)
Để đảm bảo tiến độ Tuần 5-8 không bị vỡ, nhóm thống nhất quy tắc sau:
1. Lịch họp nhóm (Sync-up):
Tần suất: ....... lần/tuần.
Thời gian cố định: Thứ ....... lúc ....... giờ.
Hình thức: [ ] Offline | [ ] Online (Meet/Zoom/Discord).
2. Quy trình quản lý Source Code (Git Workflow):
Nhánh chính (Production): main hoặc master.
Nhánh phát triển (Development): dev.
Nhánh tính năng: feature/ten-tinh-nang
Quy tắc đặt tên Commit: (VD: [Feature] Login screen, [Fix] Button error...).
Quy tắc Merge: Không ai được tự ý merge vào main khi chưa được Review.
HƯỚNG DẪN QUY TRÌNH LÀM VIỆC NHÓM VỚI GIT (GIT WORKFLOW)
Để đảm bảo source code không bị xung đột và mất mát trong quá trình làm việc, yêu cầu tất cả thành viên trong nhóm tuân thủ nghiêm ngặt quy trình dưới đây.
1. Cấu trúc các nhánh (Branch Strategy)
Hệ thống của chúng ta sẽ vận hành trên 3 loại nhánh chính (như hình ảnh quy định):
🟢 1. Nhánh main (hoặc master) - Production
Chức năng: Chứa mã nguồn ổn định nhất, không có lỗi, sẵn sàng để demo hoặc nộp bài.
Quy tắc bất di bất dịch: KHÔNG ĐƯỢC code trực tiếp trên nhánh này. Chỉ được cập nhật code vào đây khi đã kiểm tra kỹ ở nhánh dev.
🟡 2. Nhánh dev - Development
Chức năng: Đây là nơi tích hợp code của cả nhóm. Tất cả các tính năng mới sau khi code xong sẽ được gộp (merge) vào đây trước.
Quy tắc: Code trên nhánh này có thể chưa hoàn hảo nhưng phải chạy được (không được crash app khi khởi động).
🔵 3. Nhánh tính năng feature/...
Chức năng: Mỗi khi bạn nhận một task, bạn phải tạo một nhánh riêng từ dev để làm việc.
Quy tắc đặt tên: feature/ten-tinh-nang-ngan-gon (viết liền, không dấu).
Ví dụ: feature/login-screen, feature/cart-logic, feature/profile-ui.
2. Quy trình làm việc hằng ngày (Daily Workflow)
Mỗi khi bắt đầu làm một tính năng mới (Ví dụ: Làm màn hình Đăng nhập), hãy thực hiện đúng 5 bước sau:
Bước 1: Cập nhật code mới nhất từ nhóm
Trước khi làm gì, hãy chắc chắn code của bạn đang đồng bộ với nhóm.
Bash
git checkout dev        # Chuyển về nhánh dev
git pull origin dev     # Tải code mới nhất của nhóm về máy
Bước 2: Tạo nhánh mới để làm việc
Tuyệt đối không code trên dev. Hãy tách nhánh riêng:
Bash
git checkout -b feature/login-screen   # Tạo và chuyển sang nhánh tính năng mới
Bước 3: Code và Commit (Lưu thay đổi)
Sau khi code xong một phần việc nhỏ, hãy commit ngay.
Quy tắc viết Commit Message: [Loại] Mô tả ngắn gọn công việc.
[Feature]: Khi làm tính năng mới. (VD: [Feature] Design UI login page)
[Fix]: Khi sửa lỗi. (VD: [Fix] Validate email error)
[Update]: Khi cập nhật nhỏ hoặc refactor code.
Bash
git add .
git commit -m "[Feature] Tao giao dien man hinh dang nhap"
Bước 4: Đẩy code lên Server (Push)
Đẩy nhánh tính năng của bạn lên GitHub/GitLab (Không đẩy thẳng vào dev).
Bash
git push origin feature/login-screen
Bước 5: Tạo yêu cầu gộp code (Pull Request / Merge Request)
Đây là bước quan trọng nhất trong mục "Quy tắc Merge".
Vào GitHub/GitLab, bạn sẽ thấy nút "Compare & pull request".
Chọn nhánh nguồn (Source): feature/login-screen.
Chọn nhánh đích (Target): dev (Lưu ý: Merge vào dev, KHÔNG merge vào main).
Gửi link PR vào nhóm chat và tag Leader hoặc người review vào kiểm tra.
3. Quy tắc Review và Merge (Dành cho Leader/Reviewer)
Theo hình ảnh quy định: "Không ai được tự ý merge vào main khi chưa được Review".
Code Review: Người được tag phải vào xem file thay đổi (Files changed).
Kiểm tra xem code có rõ ràng không?
Có đặt tên biến linh tinh không?
Có file rác không?
Approve & Merge:
Nếu ổn: Nhấn Merge (Gộp code vào dev).
Nếu chưa ổn: Comment yêu cầu sửa lại (Request changes).
4. Cách xử lý xung đột (Conflict) - Đừng hoảng!
Nếu khi Merge báo Conflict, nghĩa là bạn và một thành viên khác cùng sửa 1 dòng code.
Tại máy cá nhân (nhánh feature của bạn), gõ: git pull origin dev.
VS Code sẽ hiện ra phần bị conflict. Bạn chọn "Accept Current Change" (lấy code của bạn) hoặc "Accept Incoming Change" (lấy code của bạn kia) hoặc sửa thủ công cho đúng cả hai.
Sau khi sửa xong, thực hiện lại Bước 3 và Bước 4 (Commit và Push lại).
⚠️ Tóm tắt các lệnh cấm (Do's and Don'ts)
ĐƯỢC LÀM (✅)
CẤM TUYỆT ĐỐI (❌)
Tạo nhánh con feature/... để code
Code trực tiếp trên nhánh main hoặc dev
Commit nhiều lần, chia nhỏ công việc
Commit 1 cục to đùng "Complete project"
Viết message: [Feature] Add home banner
Viết message: "abc", "fix", "update", "xong roi"
Merge vào dev để test trước
Merge thẳng nhánh feature vào main
3. Quản lý công việc:
Công cụ theo dõi task: [ ] Trello | [ ] Notion | [ ] GitHub Projects | [ ] Excel.
Deadline cho mỗi task cá nhân là trước buổi họp nhóm ....... tiếng.
Chào bạn, tiếp nối quy trình làm việc nhóm ở phần trước, đây là Hướng dẫn chi tiết cách sử dụng các công cụ Quản lý công việc (Task Management) để nhóm của bạn không bị rơi vào cảnh "cha chung không ai khóc" hoặc trễ deadline.
Dưới đây là hướng dẫn cho 4 công cụ phổ biến nhất mà các nhóm sinh viên thường dùng: Trello, GitHub Projects, Notion, và Excel.
NGUYÊN TẮC VÀNG KHI QUẢN LÝ TASK (ÁP DỤNG CHO MỌI CÔNG CỤ)
Trước khi đi vào chi tiết công cụ, cả nhóm phải thống nhất 3 luật bất biến sau:
Rõ người - Rõ việc: Một task chỉ có 1 người chịu trách nhiệm chính (Assignee). Nếu task quá lớn, hãy chia nhỏ ra.
Cập nhật Real-time: Làm xong là kéo thẻ/đổi trạng thái ngay, không đợi đến họp mới báo cáo.
Quy tắc Deadline: Deadline trên công cụ phải được đặt trước buổi họp nhóm ít nhất 12-24 tiếng (để Leader có thời gian review và merge code).
LỰA CHỌN 1: TRELLO (Dễ dùng nhất - Khuyên dùng)
Phù hợp với: Nhóm thích sự đơn giản, trực quan, kéo thả.
Bước 1: Thiết lập Bảng (Board Structure)
Tạo một Board mới và chia thành 5 cột (List) theo luồng công việc chuẩn:
Backlog (Kho việc): Chứa tất cả ý tưởng, tính năng cần làm trong cả dự án (nhưng chưa làm ngay).
To Do (Cần làm tuần này): Leader sẽ kéo các task từ Backlog sang đây vào đầu mỗi tuần (Sprint Planning).
Doing (Đang làm): Thành viên khi bắt đầu làm việc gì thì kéo thẻ của mình sang đây.
Review/Testing: Làm xong thì kéo sang đây để Leader/Tester kiểm tra.
Done (Hoàn thành): Chỉ khi đã được duyệt/merge code thành công mới được nằm ở đây.
Bước 2: Quy định nội dung thẻ (Card Detail)
Trong mỗi thẻ công việc (Card), bắt buộc phải điền đủ:
Members: Tag tên người làm.
Labels (Nhãn màu): Phân loại (VD: Màu đỏ = Bug, Màu xanh = Feature, Màu vàng = Design).
Due Date: Ngày hết hạn (Nhớ bật nhắc nhở trước 1 ngày).
Checklist: Các đầu việc nhỏ bên trong (VD: Task "Làm màn hình Login" sẽ có checklist: Cắt UI, Bắt sự kiện nút bấm, Call API).
LỰA CHỌN 2: GITHUB PROJECTS (Tốt nhất cho Dân Code)
Phù hợp với: Dự án phần mềm, muốn quản lý code và task ở cùng một chỗ.
Bước 1: Tạo Project
Vào tab Projects trên GitHub Repository của nhóm -> Chọn New Project.
Chọn Template "Board" (dạng bảng giống Trello).
Bước 2: Liên kết với Issues (Quan trọng)
Thay vì tạo thẻ thủ công, hãy tạo Issues cho mỗi tính năng/bug:
Vào tab Issues -> New Issue -> Mô tả task cần làm.
Bên cột phải, mục Projects, chọn Project vừa tạo.
Issue này sẽ tự động xuất hiện trên bảng Project.
Bước 3: Tự động hóa (Automation)
GitHub có tính năng cực hay là Workflows:
Khi bạn tạo Pull Request và link với Issue -> Thẻ tự động nhảy sang cột Doing hoặc Review.
Khi Pull Request được Merge -> Thẻ tự động nhảy sang cột Done và Issue tự đóng lại (Close).
Cách làm: Trong Project Settings -> Workflows -> Bật "Item added to project", "Code review approved", "Pull request merged".
LỰA CHỌN 3: NOTION (Đa năng - Tài liệu + Task)
Phù hợp với: Nhóm muốn viết cả tài liệu báo cáo, kịch bản test và quản lý task cùng lúc.
Bước 1: Tạo Database
Gõ lệnh /board để tạo một bảng Kanban view.
Thêm các cột (Properties) quan trọng:
Status: To Do, In Progress, Done.
Assign: Chọn người thực hiện (Person).
Date: Deadline.
Priority: High, Medium, Low.
Bước 2: Tận dụng các View khác nhau
Board View: Để xem trạng thái task (giống Trello).
Calendar View: Để xem lịch deadline của cả tháng.
Table View: Để lọc ra xem "Tuần này tôi phải làm gì?".
LỰA CHỌN 4: GOOGLE SHEETS / EXCEL (Cổ điển)
Phù hợp với: Nhóm không rành công nghệ, muốn quản lý nhanh.
Tuy đơn giản nhưng dễ bị rối. Hãy kẻ bảng với các cột bắt buộc sau: 
| STT | Tên Task (Cụ thể) | Người làm | Trạng thái (Drop-down) | Deadline | Link sản phẩm (Github/Figma) | Ghi chú | 
| 1 | Thiết kế DB | Nam | ✅ Done | 10/10 | [Link] | | | 2 | Code API Login | Hùng | 🚧 Doing | 12/10 | [Link] | Đang vướng lỗi Auth |
Mẹo: Sử dụng Conditional Formatting (Định dạng có điều kiện) để tô màu đỏ các ô quá hạn Deadline.

MẪU BÁO CÁO GIẢ LẬP THAM KHẢO
PHIẾU BÁO CÁO TIẾN ĐỘ & PHÂN CÔNG "FULL-STACK"
Dự án: Hệ sinh thái Tin tức TLU (TLU News Ecosystem)
Công nghệ: Backend/Web (Laravel + MySQL) - Mobile (Flutter)
I. XÁC ĐỊNH DANH SÁCH MÀN HÌNH (SCOPE OF WORK)
Để đảm bảo khối lượng công việc đủ cho 3 thành viên trong 2 tháng, chúng ta xác định các màn hình chi tiết như sau:
1. Phân hệ Web Portal (Laravel Blade + MySQL)
Hệ thống Web đóng hai vai trò: Trang tin tức cho người dùng PC và Trang quản trị (CMS).
Nhóm Frontend (Người dùng xem tin):
Trang chủ (Home): Hiển thị tin nổi bật (Slider), tin mới nhất, tin theo danh mục.
Trang Danh mục (Category): List tin tức lọc theo Khoa/Phòng ban.
Trang Chi tiết tin (Detail): Nội dung bài viết, hiển thị bình luận, form gửi bình luận.
Trang Tìm kiếm (Search): Kết quả tìm kiếm theo từ khóa.
Nhóm Backend (Admin - Quản trị viên):
Admin Login: Đăng nhập quản trị.
Dashboard: Thống kê tổng số bài viết, tổng lượt xem, tổng bình luận.
Quản lý Tin tức (Post CRUD): Danh sách tin, Thêm/Sửa/Xóa tin (Tích hợp CKEditor để soạn thảo).
Quản lý Danh mục (Category CRUD): Thêm/Sửa/Xóa chuyên mục.
Duyệt Bình luận (Comment Moderation): Xem danh sách bình luận, Xóa bình luận xấu.
Quản lý Tài khoản (User Manager): Danh sách SV/GV (Reset mật khẩu nếu cần).
2. Phân hệ Mobile App (Flutter)
Ứng dụng dành cho sinh viên/giảng viên đọc tin nhanh và tương tác.
Màn hình Login: Đăng nhập bằng Email trường/Mã SV.
Màn hình Home (News Feed): Tab tin mới, Tin Hot, kéo để refresh.
Màn hình Drawer/Menu: Thanh menu trượt ngang để chọn Khoa/Phòng ban.
Màn hình List tin theo danh mục: Hiển thị danh sách khi chọn menu.
Màn hình Chi tiết tin: Đọc nội dung (Render HTML từ Laravel trả về), xem danh sách comment.
Màn hình Soạn bình luận: Input nhập comment gửi lên server.
Màn hình Thông báo: Danh sách thông báo đẩy (Notification).
Màn hình Cá nhân (Profile): Thông tin SV, Đổi mật khẩu, Tin đã lưu (Bookmark).
II. CHIẾN LƯỢC PHÂN CÔNG: "VERTICAL SLICING" (CẮT DỌC)
Yêu cầu môn học: Tất cả thành viên đều phải Thiết kế - Code - Test.
Giải pháp: Chia dự án thành 3 Module lớn. Mỗi người thầu trọn gói 1 Module.
Thành viên 1: Module Hệ thống & Người dùng (System & User)
Phạm vi: Quản lý Tài khoản, Xác thực (Auth), Cá nhân (Profile).
Nhiệm vụ cụ thể:
DB: Thiết kế bảng users, roles.
Laravel: Code Login API (Passport/Sanctum), Trang Admin quản lý User.
Flutter: Code màn hình Login, Màn hình Profile, Xử lý lưu Token đăng nhập.
Test: Test luồng đăng nhập, bảo mật API.
Thành viên 2: Module Nội dung (Content Core)
Phạm vi: Quản lý Tin tức, Danh mục, Hiển thị tin.
Nhiệm vụ cụ thể:
DB: Thiết kế bảng posts, categories.
Laravel: Code CRUD Tin tức (Admin), API lấy danh sách tin (Home/Category/Detail).
Flutter: Code màn hình Home, Menu Danh mục, Màn hình Chi tiết tin (Render HTML).
Test: Kiểm tra hiển thị bài viết, format ảnh, text.
Thành viên 3: Module Tương tác & Thống kê (Interaction & Stats)
Phạm vi: Bình luận, Tìm kiếm, Dashboard, Bookmark.
Nhiệm vụ cụ thể:
DB: Thiết kế bảng comments, bookmarks.
Laravel: Code trang Admin duyệt Comment, API Gửi/Lấy Comment, API Tìm kiếm, Trang Dashboard thống kê.
Flutter: Code chức năng Comment, Màn hình Search, Chức năng Lưu tin (Local Storage).
Test: Test chức năng search, spam comment.
III. KIẾN TRÚC CODE & TỔ CHỨC FILE
Để tránh việc 3 người sửa chung 1 file gây conflict, chúng ta quy hoạch cứng cấu trúc thư mục.
1. Phía Server (Laravel)
Sử dụng mô hình MVC chuẩn, nhưng tách Controller theo API và Web Admin riêng.
Plaintext
app/
├── Models/ (User, Post, Category, Comment)
├── Http/
│   ├── Controllers/
│   │   ├── Admin/         <-- Code cho Web Admin
│   │   │   ├── DashboardController.php (Thành viên 3)
│   │   │   ├── PostController.php      (Thành viên 2)
│   │   │   └── UserController.php      (Thành viên 1)
│   │   └── Api/           <-- Code API trả về cho Flutter
│   │       ├── AuthController.php      (Thành viên 1)
│   │       ├── NewsController.php      (Thành viên 2)
│   │       └── InteractionController.php (Thành viên 3)
routes/
├── web.php (Route cho trang Admin)
└── api.php (Route cho Mobile App)
2. Phía Mobile (Flutter)
Sử dụng cấu trúc Feature-based (Chia thư mục theo tính năng). Ai làm tính năng nào thì chỉ sửa trong thư mục đó.
Plaintext
lib/
├── main.dart (File chạy chính - Leader setup)
├── core/ (Các file dùng chung: Constants, API Client, Theme)
├── features/
│   ├── auth/              <-- Thành viên 1 làm việc ở đây
│   │   ├── screens/ (LoginScreen, ProfileScreen)
│   │   ├── services/ (AuthService - Gọi API Laravel)
│   │   └── widgets/
│   ├── news/              <-- Thành viên 2 làm việc ở đây
│   │   ├── screens/ (HomeScreen, DetailScreen)
│   │   ├── services/ (NewsService)
│   │   └── widgets/ (NewsCard, CategoryChip)
│   └── interaction/       <-- Thành viên 3 làm việc ở đây
│       ├── screens/ (CommentSection, SearchScreen)
│       ├── services/
│       └── widgets/
IV. QUY TRÌNH GIT & ĐẶT TÊN NHÁNH (RẤT QUAN TRỌNG)
Để 3 người cùng code song song mà khi gộp lại không bị lỗi, quy ước đặt tên nhánh như sau:
Cấu trúc tên nhánh: feature/[Tên-Module]/[Tên-Chức-năng]
Thành viên 1 (Auth):
feature/auth/login-api (Làm Laravel API Login)
feature/auth/flutter-ui (Làm Flutter UI Login)
Thành viên 2 (News):
feature/news/crud-laravel (Làm trang Admin thêm sửa xóa tin)
feature/news/flutter-home (Làm màn hình Home App)
Thành viên 3 (Interaction):
feature/inter/comment-system (Làm chức năng bình luận)
feature/inter/search-ui (Làm màn hình tìm kiếm)
Quy tắc Merge:
Mỗi khi code xong 1 chức năng nhỏ (VD: xong API Login), tạo Pull Request vào nhánh dev.
Leader (hoặc chéo cánh) review code. "Bạn làm Module Auth review code cho Module News" để hiểu hệ thống của nhau.
